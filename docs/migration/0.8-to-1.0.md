# Migration guide: 0.8.0-rc.1 → 1.0.0

The 1.0.0 release intentionally minimises breaking changes from the RC.
Most users can upgrade by replacing the library binary and recompiling —
no source changes required.

## Breaking changes

### 1. `HELIX_ABI_VERSION` encoding changed

**Old (0.8.0-rc.1):** encoded as `0xMMmmPP` (major × 65536 + minor × 256 + patch).  
**New (1.0.0):** encoded as `(major × 65536 + minor)` — patch is implicit.

| Version | Old value | New value |
|---------|-----------|-----------|
| 0.1.0 | `0x000100` | n/a |
| 1.0.0 | n/a | `0x00010000` |

**Impact:** code that compares `helix_abi_version()` against a hard-coded
integer needs updating. Code that calls `helix_abi_version()` at runtime for
logging purposes is unaffected.

**Fix:**
```c
/* Old */
assert(helix_abi_version() == 0x000100);

/* New */
assert(helix_abi_version() == 0x00010000);
/* or, robustly: */
assert((helix_abi_version() >> 16) == 1 &&
        helix_abi_version() >= HELIX_ABI_VERSION);
```

### 2. Linux versioned symbol node renamed

The GNU linker version node changed from `HELIX_0.1` to `HELIX_1.0`.

**Impact:** only affects binaries that were linked against the RC and are
now runtime-loaded against 1.0.0 without relinking. In practice this means
you must relink (not just rerun) any binary that was built against the RC.

**Fix:** recompile and relink against `libhelix.so.1`. This is the expected
upgrade path for a major-version release.

### 3. `helix_version_string()` return value

Returns `"1.0.0+llama.cpp-<sha>"` instead of `"0.1.0+llama.cpp-<sha>"`.

**Impact:** only code that parses the version string (unusual) needs updating.

### 4. SOVERSION 0 → 1

The shared library file changed from `libhelix.so.0.x.y` to `libhelix.so.1.0.0`.
The soname is now `libhelix.so.1`.

**Impact:** any `-rpath` or `LD_LIBRARY_PATH` configuration pointing at a
directory containing only the old `libhelix.so.0` will need to be updated
once the 1.0.0 artefact is installed.

---

## Non-breaking changes that may affect behaviour

- `helix.pc` description string changed from "Embeddable, OpenAI-isomorphic
  local LLM library" to "OpenAI-isomorphic on-device LLM library". Build
  scripts that `pkg-config --modversion helix` will now see `1.0.0`.
- The `libdir` and `includedir` variables in `helix.pc` now use
  `CMAKE_INSTALL_LIBDIR` / `CMAKE_INSTALL_INCLUDEDIR` (GNUInstallDirs), which
  on multiarch Debian systems resolves to e.g. `lib/x86_64-linux-gnu` rather
  than `lib`. Downstream CMake consumers using `find_package(helix)` are
  unaffected; consumers parsing `helix.pc` directly should use
  `pkg-config --variable=libdir helix`.

---

## Upgrade steps

1. Replace `libhelix.so.0.*` with `libhelix.so.1.*` (or install via package
   manager: `brew upgrade helix`, `apt-get upgrade libhelix1`).
2. Update any hard-coded `HELIX_ABI_VERSION` comparisons (see above).
3. Relink your binary against `libhelix.so.1` (the soname changed).
4. If using `find_package(helix)` in CMake, bump the required version:
   ```cmake
   find_package(helix 1.0 REQUIRED)   # was: find_package(helix 0.1 REQUIRED)
   ```
5. Run `helix-doctor` to confirm the installed library reports `1.0.0`.

That is the complete list. All 16 function signatures, all JSON request and
response shapes, and all error code integer values are identical between the
RC and 1.0.0.
